# Bounty Plaza Rules — 质量竞速制

## 核心原则

> **代码质量优先，同质量先到先得。**
> 谁的代码质量好就用谁的；质量一样好时，谁先提交谁赢。

---

## 任务生命周期

### 并行流程

```
发现 GitHub 赏金任务
       |
       +---> 对外: AI 生成初版修复 -> 提 PR (锁定赏金)
       |
       +---> 对内: 在 bounty-plaza 发布任务 Issue
                              |
                    贡献者提交修复 -> 自动评分 (>=90)
                              |
                    第一个达标的提交者获胜
                              |
                    force-push 获胜代码到占位 PR
                              |
                    上游合并 PR -> 赏金到管理员账户
                              |
                    发放积分币给获胜者
```

### 分步说明

1. **发现任务** → 管理员找到 GitHub 上的赏金 Issue
2. **占坑（对外）** → AI 在 **5 分钟内**生成一版初步修复，然后 fork 上游仓库、提交 PR
   - PR 标题写明修复的 Issue 编号
   - PR 内容是一份能编译、能跑的正经修复代码
   - 维护者看到会认为"有人在做了"，不会关闭
   - **绝不能提空 PR**——空 PR 会被维护者关闭并标记 spam
3. **分发（对内）** → 同时在 bounty-plaza 创建任务 Issue，写明漏洞描述、上游赏金额、积分币标价、评分规则
4. **竞优** → 贡献者提交修复 → 自动评分 ≥90 分 → 第一个达标的获胜
5. **覆盖** → 管理员用获胜代码 force-push 到占位 PR，更新 PR 描述为最终修复说明
6. **等待上游** → 上游审核 PR，可能需要迭代修改
7. **结算** → **上游合并 PR → 赏金到管理员账户 → 才发放积分币给获胜者**
8. **兑换** → 贡献者可通过积分币系统兑换现金

> ⚠️ 积分币仅在上游赏金确认到账后发放。评分达标≠一定能拿到积分币。

## 提交规则

- 所有修复必须通过 **Pull Request** 提交到 bounty-plaza
- 每个 PR 必须包含：
  - ✅ 能工作的修复代码
  - ✅ 修复说明
  - ✅ 测试用例
- 管理员触发评分流程

## 评分规则（百分制）

### 评分维度

| 维度 | 权重 | 满分 | 评分方式 |
|------|:----:|:----:|----------|
| 🧪 **功能正确性** | 40% | 40 | pytest 全量测试。全部通过 = 40 分；有任何失败 = 一票否决 0 分 |
| 🔒 **安全性** | 35% | 35 | AST 静态分析 + bandit 安全扫描。检测到 1 项违规扣 7 分，扣完为止 |
| 📐 **代码质量** | 15% | 15 | pylint 评分 / 10 × 15。圈复杂度超阈值每个扣 1 分，最多扣 5 分 |
| ⚡ **性能** | 10% | 10 | 执行时间 ≤ 基线 ×1 = 10 分；≤ ×2 = 8 分；≤ ×5 = 5 分；> ×5 = 2 分 |

### 一票否决项（总分直接 0 分）

| 违规行为 | 检测方式 | 说明 |
|----------|----------|------|
| 直接 return 预期输出 | AST 分析 | 检测到函数直接返回硬编码字符串/列表 |
| 删除/清空测试用例 | 文件 hash 校验 | 测试文件被篡改或行数 < 3 |
| 使用禁止模块 | import 白名单 | pickle / marshal / ctypes / eval / exec |
| 危险系统调用 | AST 分析 | os.system / subprocess.Popen 等 |
| 代码为空/乱码 | 长度 + AST | 代码长度 < 10 或 AST 解析失败 |
| 高危安全漏洞 | bandit | bandit HIGH/MEDIUM 级别告警 |

### 达标线

| 总分 | 结果 |
|:----:|:----:|
| **≥ 90 分** | ✅ 达标，可进入排名 |
| < 90 分 | ❌ 未达标，可修改后重新提交 |

## 判定规则

1. 系统自动评分（0-100）
2. **质量分 ≥ 90 分的提交才有资格获胜**
3. 从达标的提交中按质量分从高到低排序
4. **质量最高者获胜**
5. 若多人质量分相同 → **先提交的获胜**
6. 获胜者确定后关闭 Issue，其他 PR 标记为已过期

## 任务标价（分账规则）

每个任务的积分币标价取决于上游赏金额度和分成比例。

### 分成比例

### 分成比例

| 上游赏金（USD） | 赢家分成 | 平台抽成 | 说明 |
|-----------------|:--------:|:--------:|------|
| ≤ $200 | **70%** | 30% | 小任务，激励优先 |
| $200 - $500 | **65%** | 35% | 中等任务 |
| $500 - $1000 | **60%** | 40% | 大任务 |
| $1000+ | **55%** | 45% | 特大任务，平台投入更多沟通精力 |
| 500+ | **50%** | 50% | 特大任务，平台投入大量精力 |

### 标价计算公式



> 每次发布 Issue 时，管理员会在描述中写明：上游赏金、分成比例、积分币标价。

### 发放条件

积分币发放需同时满足以下条件：
1. ✅ 提交者代码评分 ≥ 90 分
2. ✅ 提交者被判定为获胜者
3. ✅ 管理员已向上游提 PR
4. ✅ **上游已合并 PR**
5. ✅ **赏金已到管理员账户**
6. ❌ 以上任一条件不满足 → 积分币暂不发放

## 积分币兑换

积分币系统使用独立 SQLite 账本，防篡改链式审计。

| 项目 | 数值 |
|------|:----:|
| 汇率 | **100 积分币 = $0.01** |
| 最低兑换 | **10,000 积分币**（=$1.00） |
| 兑换方式 | 运行 `python scripts/coin.py redeem --user <name> --amount <n> --address <paypal/usdt>` |
| 处理时间 | 管理员审核后 7 个工作日内打款 |
| 审计 | 每笔交易链式哈希，不可篡改 |

### 积分币命令

```bash
# 查余额
python scripts/coin.py balance <用户名>

# 查看排行榜
python scripts/coin.py ledger

# 发起兑换
python scripts/coin.py redeem --user <用户名> --amount 10000 --address "PayPal: user@example.com"

# 管理: 批准兑换
python scripts/coin.py approve --id <请求编号>

# 管理: 标记已打款
python scripts/coin.py pay --id <请求编号>

# 审计日志
python scripts/coin.py audit
```

## 时间窗口

- 任务发布后**立即开放**提交
- 获胜者产生后**立即停止**接收新提交
- 若 **72 小时**无人达标 → 标记为 `unclaimed`

## 禁止行为

- ❌ 抄袭他人提交
- ❌ 提交恶意代码
- ❌ 多个账号参与同一任务
- ❌ 提交不相关的代码
- ❌ 篡改测试用例骗取评分

---

> 💡 质量优先，反作弊零容忍。修得好 + 修得快 + 不作弊 = 赢家。
